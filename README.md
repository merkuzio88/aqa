# Тестовое задание: QA Automation (API)

Автоматизированное тестирование веб-сервиса с использованием мок-сервера для имитации внешних зависимостей.

📊 **Allure Отчет:** [Посмотреть актуальный отчет](https://merkuzio88.github.io/aqa/)

## 🛠 Стек технологий
* **Java 17**
* **JUnit 5** — тестовый фреймворк
* **RestAssured** — HTTP-клиент для API тестов
* **WireMock** — мокирование внешних сервисов (`/auth`, `/doAction`)
* **Allure Report** — генерация понятных отчетов для нетехнических специалистов
* **Maven** — сборка проекта
* **GitHub Actions & GitHub Pages** — CI/CD пайплайн и хостинг отчетов

## 🏗 Архитектурные решения во фреймворке
1. **Безопасность (SecOps):** В репозитории отсутствуют бинарные файлы (`.jar`) и чувствительные данные (пароли, ссылки на инфраструктуру). В CI/CD пайплайне приложение скачивается и распаковывается динамически с использованием **GitHub Secrets**.
2. **Изоляция:** Перед каждым тестом конфигурация мок-сервера сбрасывается (`WireMock.reset()`), а для авторизации генерируется уникальный токен. Тесты не влияют друг на друга.
3. **Configuration Management:** URL, порты и статические ключи вынесены в файл `application.properties`.
4. **Data-Driven Testing:** Использование параметризованных тестов (`@ValueSource`) для проверки граничных значений и форматов валидации.

## ⚙️ CI/CD и запуск в облаке
В проекте настроен автоматизированный пайплайн (**GitHub Actions**):
1. **Автозапуск:** Тесты прогоняются автоматически при каждом `push` в ветку `main`.
2. **Ручной запуск:** Реализована поддержка `workflow_dispatch`. Запустить тесты можно вручную во вкладке **Actions** -> **API Tests CI** -> **Run workflow**. Это позволяет проводить Smoke-тестирование без изменения кода.
3. **Безопасность:** Ссылка на дистрибутив и пароль к архиву не хранятся в коде, а передаются через **GitHub Secrets**.
4. **Отчетность:** После завершения тестов Allure-отчет автоматически публикуется на **GitHub Pages**.

## 🚀 Инструкция по запуску локально

### 1. Подготовка
Скачайте архив с приложением (`internal-0.0.1-SNAPSHOT.jar`) согласно ТЗ и положите его в корень данного проекта.

### 2. Запуск тестируемого приложения (Docker)
Для изоляции и корректной работы сети с мок-сервером рекомендуется запускать приложение в Docker.
В корне проекта выполните:
```bash
docker run -d --rm -p 8080:8080 -v ${PWD}:/app -w /app eclipse-temurin:17-jre java -jar -Dsecret=qazWSXedc -Dmock=http://host.docker.internal:8888/ internal-0.0.1-SNAPSHOT.jar
```
*(Для Linux замените `host.docker.internal` на `localhost` и добавьте `--network host`)*

### 3. Запуск тестов
В корне проекта выполните команду:
```bash
mvn clean test
```

### 3. Просмотр отчета Allure
После прогона тестов сгенерируйте отчет:
```bash
mvn allure:serve
```
Отчет автоматически откроется в браузере. В нем описаны подробные шаги (Steps) на русском языке, а также прикреплены логи запросов и ответов.

## 🐞 Заметки по тестированию (Найденный дефект)

В ходе автоматизации было выявлено несоответствие поведения приложения и требований документации.

**Описание:**
В ТЗ указано, что `token` — это строка длиной 32 символа, состоящая из `A-Z0-9`.
Однако, при отправке токена, содержащего буквенные символы (например, `USER1234567890123456789012345678`), приложение возвращает ошибку `400 Bad Request`. Приложение успешно принимает только токены, состоящие исключительно из цифр `0-9`.

**Workaround в тестах:**
Чтобы избежать падения End-to-End тестов по причинам, не связанным с проверяемой бизнес-логикой (Workflow), генератор случайных токенов в классе BaseTest.java временно перенастроен на генерацию валидных токенов только из цифр (0-9). Ошибка валидации формата покрыта отдельным параметризованным тестом в классе NegativeTest.
